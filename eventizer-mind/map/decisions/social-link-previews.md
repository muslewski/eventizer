# Social link previews (Open Graph images)

## Context
Sharing an Eventizer link (the homepage `www.eventizer.pl`, an offer page) on Instagram,
Messenger, WhatsApp, iMessage, Slack, Discord, etc. often showed **no preview image** — sometimes
working, sometimes not. Those platforms all scrape Open Graph tags via `facebookexternalhit` and
friends. Investigation found three compounding root causes in `src/utilities/generateMeta.ts` +
`mergeOpenGraph.ts` (both inherited largely unchanged from the Payload website template):

1. **The static default image was a 404.** The fallback `og:image` pointed at
   `/website-template-OG.webp`, but that file **never existed** in `public/` (only `my-favicon/`
   was there). Any page without its own SEO image — the homepage included — emitted an `og:image`
   that 404s, so the crawler had nothing to render.
2. **The origin was prepended onto already-absolute upload URLs.** Uploads live on Vercel Blob
   (`vercelBlobStorage`, `clientUploads: true`), so a media/offer `url` is already an absolute
   `https://<store>.public.blob.vercel-storage.com/…`. The old code did `serverUrl + image.url`
   unconditionally, producing a **malformed** `https://www.eventizer.plhttps://…blob…` URL.
   So every page that *did* have an image (offers, pages with an SEO image) also failed.
3. **WebP format.** Offer images are stored as WebP (`OfferUploads.formatOptions.format = 'webp'`),
   and crawlers render WebP inconsistently across clients — a third, format-level flake.

## Decision
- Ship a **real, crawler-safe branded default** at `public/og-image.png` (1200×630 PNG, dark brand
  background + the dark-variant logo + copper accent), regenerated by
  `scripts/og/generate-default-og.mjs`. PNG, not WebP, because every crawler renders it.
- In `generateMeta`, resolve a doc's image with a `resolveOgImage` helper that **only prepends the
  origin for relative paths** (absolute `http(s)` URLs pass through untouched), forwards
  `width`/`height`/`type` when known (explicit dimensions make the *first* scrape render instead of
  skipping), and falls back to the default PNG.
- `mergeOpenGraph` default points at the PNG with Polish copy + `locale: 'pl_PL'`; `generateMeta`
  also emits a `twitter: { card: 'summary_large_image' }` block.
- The frontend root layout sets `metadataBase = new URL(getServerSideURL())` so relative metadata
  (icons, fallbacks) resolves to the real origin and Next's build warning goes away.

## Why
Crawlers silently drop a missing or malformed `og:image`, and the homepage had no SEO image set, so
the most-shared link failed every time — matching the "sometimes broken" report (it broke wherever
no valid image resolved). Keeping absolute Blob URLs intact fixes images site-wide; explicit
dimensions fix the first-scrape race that otherwise needs a manual re-scrape.

## Consequences
- Any new metadata code path must **not** prefix the server origin onto absolute upload URLs.
- `public/og-image.png` must stay a real, committed, non-WebP file; regenerate via the script if
  branding changes.
- Offer `mainImage` is still served as WebP. The `og:image` URL is now valid and modern Facebook
  renders WebP, so previews work — but for maximum compatibility with older/edge clients a JPEG
  `og` image size (or a dynamic transcode) is the next step → see the tech-debt note
  `og-image-webp-for-offers`.
